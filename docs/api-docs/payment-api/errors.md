# Ошибки

PayFlow возвращает ошибки в едином формате, независимо от эндпоинта.
Эта страница — единственный источник правды для кодов ошибок: остальные
страницы документации ссылаются сюда, а не дублируют таблицы.

## Формат ошибки

```json
{
  "error": {
    "type": "card_error",
    "code": "card_declined",
    "message": "Карта отклонена эмитентом.",
    "decline_code": "insufficient_funds",
    "request_id": "req_9F3kLp2Q"
  }
}
```

| Поле | Описание |
|---|---|
| `type` | Категория ошибки (см. таблицу ниже) — используйте для программной обработки в коде |
| `code` | Конкретный код ошибки — для более точной обработки и для логов |
| `message` | Человекочитаемое описание — **не парсите его в коде**, оно может меняться |
| `decline_code` | Только для `card_error`: причина отказа со стороны эмитента карты |
| `request_id` | Идентификатор запроса — указывайте его в обращениях в поддержку PayFlow |

!!! tip "Почему `message` не для парсинга"
    Текст `message` мы можем менять и переводить без предупреждения —
    это часть UX, а не API-контракта. Контракт — это `type` и `code`.
    Эта оговорка — то, что чаще всего забывают указать в документации
    платёжных API, а потом получают сломанные интеграции при любом
    изменении формулировки.

## HTTP-статусы

| Статус | Значение |
|---|---|
| `200 OK` | Запрос выполнен успешно |
| `400 Bad Request` | Некорректные параметры запроса |
| `401 Unauthorized` | Неверный или отсутствующий API-ключ |
| `402 Payment Required` | Запрос синтаксически верен, но платёж отклонён (см. `card_error`) |
| `404 Not Found` | Объект с таким ID не существует в этом режиме (test/live) |
| `409 Conflict` | Конфликт `Idempotency-Key` с другими параметрами запроса |
| `429 Too Many Requests` | Превышен лимит запросов — см. ниже |
| `500 / 503` | Ошибка на стороне PayFlow — повторите запрос с экспоненциальной паузой |

## Типы ошибок (`type`)

| `type` | Когда возникает | Что делать |
|---|---|---|
| `card_error` | Карта отклонена эмитентом или банком | Показать пользователю понятную причину, предложить другую карту |
| `validation_error` | Не хватает обязательного параметра или неверный формат | Ошибка на стороне интеграции — проверить запрос по [референсу параметров](../generated-reference/index.md) |
| `authentication_error` | Неверный ключ, ключ не того режима (test/live) | Проверить, что ключ существует и относится к нужному режиму |
| `idempotency_error` | Тот же `Idempotency-Key`, но другие параметры запроса | Использовать новый ключ идемпотентности для нового запроса |
| `rate_limit_error` | Превышен лимит запросов в секунду | См. раздел «Лимиты» ниже |
| `api_error` | Внутренняя ошибка PayFlow | Повторить запрос; если повторяется — обратиться в поддержку с `request_id` |

## Частые `decline_code` для `card_error`

| `decline_code` | Типичная причина | Рекомендация для пользователя |
|---|---|---|
| `insufficient_funds` | Недостаточно средств на счёте | Предложить другой способ оплаты |
| `card_declined` | Общий отказ банка без детализации | Предложить другую карту или обратиться в банк |
| `expired_card` | Истёк срок действия карты | Попросить обновить данные карты |
| `incorrect_cvc` | Неверный код CVC | Попросить ввести данные карты повторно |
| `processing_error` | Временная ошибка на стороне платёжной сети | Предложить повторить попытку через некоторое время |

## Лимиты запросов (rate limits)

По умолчанию: **100 запросов в секунду** на аккаунт в боевом режиме,
**25 запросов в секунду** в тестовом. При превышении PayFlow возвращает
`429` с заголовком `Retry-After` (в секундах).

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 2
```

Рекомендуемая стратегия повторов — экспоненциальная задержка с джиттером:

```python
import time
import random

def request_with_retry(fn, max_attempts=5):
    for attempt in range(max_attempts):
        response = fn()
        if response.status_code != 429:
            return response
        delay = min(2 ** attempt + random.random(), 30)
        time.sleep(delay)
    raise RuntimeError("Превышено число попыток")
```
