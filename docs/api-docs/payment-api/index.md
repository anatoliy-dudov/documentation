# PayFlow API — аннотированная документация

PayFlow — вымышленный платёжный API, документация к которому написана
вручную (в отличие от [генерируемого референса TaskFlow](../generated-reference/index.md)).
Пример моделирует то, как обычно устроена документация популярных платёжных
провайдеров: раздел для быстрого старта, аутентификация, единый справочник
ошибок и вебхуки.

!!! note "Для ревьюера"
    В этом разделе я специально оставляю больше «объяснений решений», чем
    было бы в реальной прод-документации — так виден ход мысли. В реальном
    проекте часть этих пояснений ушла бы в ADR (architecture decision record)
    или в комментарии к pull request, а сама документация была бы суше.

## Структура раздела

| Страница | Что закрывает | Основная аудитория |
|---|---|---|
| [Быстрый старт](quickstart.md) | Путь от регистрации до первого успешного платежа за 5 шагов | Разработчик, который делает первую интеграцию |
| [Аутентификация](authentication.md) | API-ключи, тестовый/боевой режим, идемпотентность | Разработчик на этапе настройки, security-ревьюер |
| [Ошибки](errors.md) | Формат ошибки, коды, что с ними делать | Разработчик, который дебажит интеграцию |
| [Вебхуки](webhooks.md) | Асинхронные уведомления о статусе платежа | Разработчик, который строит бэкенд-обработку событий |

## Принципы, которые я использовал при написании

1. **Задача → концепция → референс.** Быстрый старт даёт задачную
   последовательность действий. Аутентификация и вебхуки объясняют
   концепции. Ошибки — справочник, к которому возвращаются точечно.
   Это разные типы чтения (`step-by-step` vs `lookup`), и смешивать их
   в одной странице вредно для обоих сценариев.
2. **Один источник правды для кодов ошибок.** Все коды ошибок описаны
   один раз, на странице [«Ошибки»](errors.md). Остальные страницы
   ссылаются на неё, а не дублируют таблицу — иначе при обновлении API
   придётся синхронизировать несколько копий.
3. **Явное разделение test/live режима.** Платёжные API — зона повышенного
   риска: одна пропущенная фраза про боевой режим может привести к
   реальному списанию денег во время тестов. Это подсвечено отдельно на
   нескольких страницах, а не один раз в самом начале, где это легко
   пропустить.
4. **Код-примеры на нескольких языках.** Технический писатель не обязан
   поддерживать SDK на 10 языков, но должен уметь показать хотя бы
   `curl` + один серверный язык (здесь — Python), чтобы пример можно было
   скопировать и сразу запустить.

## Что это НЕ показывает

Честно обозначаю границы примера:

- Это не настоящий платёжный провайдер — все идентификаторы, суммы,
  ключи и параметры вымышленные.
- Раздел не покрывает полный жизненный цикл (возвраты, диспуты,
  мультивалютность, PCI DSS compliance) — это увеличило бы объём в разы
  без дополнительной пользы для целей портфолио.
