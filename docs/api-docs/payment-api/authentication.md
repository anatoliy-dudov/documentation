# Аутентификация

PayFlow использует API-ключи и HTTP Basic Auth: ключ передаётся как логин,
пароль оставляется пустым.

```bash
curl https://api.payflow.example/v1/payment_intents \
  -u pf_test_sk_51H8x...:
```

## Тестовый и боевой режим

У каждого аккаунта есть два независимых режима — **test** и **live** — и
у каждого своя пара ключей (публичный + секретный). Данные между режимами
не пересекаются: платёж, созданный тестовым ключом, никогда не появится
в боевой выписке.

!!! danger "Самая частая ошибка интеграции"
    Использование `pf_live_sk_...` в коде, который выполняется в браузере
    (React, Vue, обычный `<script>`). Секретный ключ виден в devtools —
    любой посетитель страницы получает полный доступ к вашему аккаунту
    PayFlow. Секретный ключ живёт **только на сервере**.

| | Публичный ключ (`pk`) | Секретный ключ (`sk`) |
|---|---|---|
| Где хранить | Клиент (браузер, мобильное приложение) | Сервер, секрет-менеджер |
| Что позволяет | Создать токен карты, инициализировать форму оплаты | Любую операцию: списания, возвраты, просмотр данных |
| Что будет при утечке | Ничего критичного — ключ для этого и публичный | Полная компрометация аккаунта — нужна немедленная ротация |

## Ротация ключей

Секретный ключ нужно ротировать:

1. немедленно — если есть подозрение на утечку (ключ засветился в
   публичном репозитории, логах, скриншоте);
2. планово — раз в 90–180 дней, по внутренней security-политике.

Порядок ротации без даунтайма:

1. Создайте новый секретный ключ в Dashboard (старый продолжает работать).
2. Разверните новый ключ во всех сервисах, которые обращаются к PayFlow.
3. Убедитесь, что трафик со старым ключом прекратился (Dashboard →
   API keys → Usage).
4. Отзовите старый ключ.

## Идемпотентность

Сетевые сбои — обычное дело: клиент отправил запрос на списание, но не
получил ответ из-за таймаута. Простой повтор запроса рискует списать
деньги дважды. Чтобы этого избежать, передавайте заголовок
`Idempotency-Key` с уникальным значением на каждую **логическую**
операцию:

```bash
curl https://api.payflow.example/v1/payment_intents \
  -u pf_test_sk_51H8x...: \
  -H "Idempotency-Key: order-8842-attempt-1" \
  -d amount=2000 -d currency=usd
```

Если запрос с тем же `Idempotency-Key` придёт повторно в течение 24 часов,
PayFlow вернёт результат первого запроса, не выполняя операцию заново.

!!! tip "Почему это объяснено здесь, а не в справочнике эндпоинтов"
    Идемпотентность — не свойство одного эндпоинта, а сквозная концепция
    для всего API. Если описать её только в справочнике параметров
    `POST /payment_intents`, разработчик, который читает про `/refunds`,
    может не узнать, что механизм тот же самый.
