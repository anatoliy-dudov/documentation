# TaskFlow API — автогенерируемый референс

Ниже — интерактивный референс, отрендеренный напрямую из файла
[`openapi.yaml`](openapi.yaml) с помощью Swagger UI. Это не переписанный
вручную текст: если изменить спецификацию, документация обновится
автоматически при следующей сборке сайта.

## Что я делал руками в этом примере

Генератор строит структуру и таблицы параметров, но не пишет за автора
смысл. Моя работа как технического писателя здесь была сосредоточена на
самой спецификации:

- **Тексты `summary` и `description`.** Например, для `DELETE /projects/{projectId}`
  явно указано, что это архивирование, а не необратимое удаление —
  без этого уточнения разработчик мог бы решить, что данные теряются
  безвозвратно, и не вызвать метод там, где он на самом деле нужен.
- **Примеры значений (`example`).** Для path-параметров и полей тела
  запроса подобраны реалистичные примеры (`proj_8f3ac1`, `task_1029ab`),
  а не `string`/`123` — это ускоряет тестирование запросов «по месту».
- **Коды ошибок и общие responses.** Вынесены в `components/responses`
  (`BadRequest`, `Unauthorized`, `NotFound`), чтобы не дублировать описание
  ошибки на каждом эндпоинте и гарантировать, что коды ошибок
  документированы единообразно.
- **Enum вместо свободной строки.** Поле `status` задано как `enum`
  (`todo`, `in_progress`, `blocked`, `done`), а не произвольная строка —
  так Swagger UI сразу показывает разработчику допустимые значения,
  и это же ограничение можно использовать для валидации.

## Инструменты и процесс

- **Линтинг спецификации** — например, [Spectral](https://github.com/stoplightio/spectral),
  чтобы ловить нарушения соглашений об именовании и отсутствующие описания
  ещё до ревью.
- **Ревью спецификации как кода** — правки в `openapi.yaml` проходят через
  pull request, как обычный код, с ревью от техписателя и от разработчика
  эндпоинта.
- **Рендеринг** — на этом сайте используется плагин MkDocs
  `mkdocs-render-swagger-plugin`, который встраивает Swagger UI прямо
  в страницу.
