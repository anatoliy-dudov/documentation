# Устраняем ошибку 429 Too Many Requests

**Кому подходит:** разработчикам, интегрирующим TaskFlow API.

**Симптом:** запросы к API периодически возвращают `HTTP 429` вместо
ожидаемого ответа, особенно при массовой синхронизации данных
(например, при первичном импорте задач).

## Почему это происходит

TaskFlow API ограничивает частоту запросов на уровне рабочего
пространства: **120 запросов в минуту** на стандартном тарифе. Лимит
общий для всех ключей одного рабочего пространства — если у вас несколько
сервисов используют один и тот же API-ключ, они делят один лимит.

Это не ошибка интеграции сама по себе — это ожидаемое поведение при
превышении лимита. Задача — обрабатывать `429` корректно, а не избежать
его полностью.

## Как быстро проверить, что дело именно в лимите

Ответ `429` содержит заголовок `Retry-After` — число секунд, через
которое можно повторить запрос:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 8
Content-Type: application/json

{"error": {"code": "rate_limited", "message": "Rate limit exceeded"}}
```

Если `Retry-After` присутствует и `error.code` равен `rate_limited` —
дело в лимите, а не в другой проблеме.

## Решение

1. **Реализуйте экспоненциальную задержку с джиттером.** Не повторяйте
   запрос сразу — используйте значение `Retry-After` как минимальную
   задержку, добавляя случайный джиттер, чтобы избежать «стада» повторных
   запросов от нескольких воркеров одновременно.
2. **Ограничьте параллелизм при массовой синхронизации.** Если вы
   импортируете тысячи задач, используйте очередь с ограниченным числом
   одновременных запросов (например, 5 параллельных запросов вместо 100).
3. **Кешируйте, где это уместно.** Если один и тот же список задач
   запрашивается несколько раз за короткий период (например, разными
   частями вашего приложения), кешируйте ответ на несколько секунд вместо
   повторного запроса к API.
4. **Для больших объёмов — используйте bulk-эндпоинты.** Там, где TaskFlow
   API предоставляет пакетные операции (`POST /tasks/bulk`), они считаются
   за один запрос к лимиту вместо N — используйте их вместо цикла из
   отдельных вызовов.

## Если лимита стандартного тарифа объективно не хватает

Лимит можно увеличить на тарифе Enterprise или обсудить индивидуальный
лимит с командой поддержки — напишите в поддержку с описанием сценария
использования (ожидаемое количество запросов в минуту и характер
нагрузки: равномерная или пиковая).
