# Настройка единого входа (SSO)

**Кому подходит:** администраторам рабочего пространства TaskFlow на
тарифе Business или Enterprise.

**Что вы получите:** сотрудники будут входить в TaskFlow через
корпоративного провайдера identity (Okta, Azure AD, Google Workspace)
вместо отдельного пароля TaskFlow.

## Перед началом

- SSO доступен на тарифах **Business** и **Enterprise**. На тарифе Free
  и Pro пункт меню SSO будет скрыт.
- Вам понадобятся права администратора и в TaskFlow, и в вашем
  identity-провайдере (IdP).
- Поддерживаемый протокол — **SAML 2.0**.

## Шаги

1. В TaskFlow откройте **Настройки рабочего пространства → Безопасность →
   Single Sign-On**.
2. Нажмите **Настроить SSO** — TaskFlow сгенерирует **Metadata URL** и
   **ACS URL**, которые понадобятся на стороне IdP.
3. В вашем IdP создайте новое SAML-приложение и укажите там значения
   из шага 2 (порядок полей отличается у разных провайдеров — см. таблицу
   ниже).
4. В IdP настройте передачу атрибута `email` — TaskFlow сопоставляет
   пользователей по email при первом входе через SSO.
5. Вернитесь в TaskFlow и вставьте **Metadata URL вашего IdP** (или
   загрузите XML-файл метаданных) в поле **Identity Provider metadata**.
6. Нажмите **Отправить тестовый вход** — TaskFlow откроет тестовое SSO-окно.
   Если вход прошёл успешно, нажмите **Включить SSO для всех**.

## Соответствие полей по провайдерам

| TaskFlow | Okta | Azure AD |
|---|---|---|
| ACS URL | Single sign-on URL | Reply URL |
| Entity ID | Audience URI (SP Entity ID) | Identifier (Entity ID) |
| Metadata URL | — (укажите вручную Entity ID и SSO URL) | App Federation Metadata Url |

## Что произойдёт с существующими паролями

После включения SSO для всех пользователей вход по обычному паролю
TaskFlow отключается для всех, кроме владельца рабочего пространства —
у владельца всегда остаётся резервный вход по паролю на случай проблем
с IdP.

## Если что-то не работает

- **«Тестовый вход» показывает ошибку `invalid_signature`** — обычно
  означает, что в TaskFlow загружены устаревшие метаданные IdP (например,
  после ротации сертификата в IdP). Повторно загрузите актуальный
  metadata-файл.
- **Пользователь входит, но получает «Доступ запрещён»** — проверьте, что
  email в атрибутах SAML совпадает с email пользователя в TaskFlow
  (регистр не важен, но домен должен совпадать).
- **SSO работает у части сотрудников, у части — нет** — проверьте, не
  назначены ли этим пользователям индивидуальные исключения в
  **Безопасность → Исключения SSO**.
