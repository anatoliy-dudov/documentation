### Standards and methodologies

I'm well versed in documentation standards relevant to the Russian/CIS
market and understand their strengths and limitations:

- **GOST 19 (ESPD)** — the Unified System of Program Documentation. These
  standards largely retain their 1970s–1980s numbering and structure and
  are advisory rather than mandatory — but remain a useful shared vocabulary
  for defining documentation scope.
- **GOST 34** — standards for automated systems, substantively updated in
  2020–2021; they map well onto modern IT system implementation projects.
- **ESKD** — the Unified System for Design Documentation, used as a
  reference where documentation concerns technical/engineering products
  rather than software alone.
- **Docs-as-code, DITA, OpenAPI/Swagger** — modern approaches to structuring
  and versioning documentation, which I combine with GOST requirements where
  it's a good fit (e.g., internal documentation for regulated industries).

### Tools

Markdown, MkDocs / MkDocs Material, Sphinx, OpenAPI/Swagger, Git/GitHub,

### How this portfolio is built

Every example in this portfolio is a standalone, complete artifact written
for demo products (**TaskFlow**, **PayFlow**), not an edited fragment of
someone else's documentation. That's intentional: it shows the full
thought process — from section structure down to specific wording — with
no ambiguity about what was copied from where.
