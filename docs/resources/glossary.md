# Глоссарий

Термины, которые используются в этом портфолио — из мира API-документации,
баз знаний и ITIL-процессов, а также стандартов ГОСТ.

## API-документация

**API (Application Programming Interface)**
: Программный интерфейс, через который одна система обращается к другой.
В контексте документации — то, что документируется: эндпоинты, параметры,
форматы ответов.

**OpenAPI (ранее Swagger)**
: Формальная спецификация для описания REST API в YAML/JSON. Служит
единым источником правды, из которого можно сгенерировать документацию,
клиентские SDK и тесты. Пример в этом портфолио — [`openapi.yaml` для TaskFlow API](../api-docs/generated-reference/openapi.yaml).

**Docs-as-code**
: Подход, при котором документация хранится в том же репозитории, что и
код, пишется в текстовом формате (Markdown, reStructuredText) и проходит
через тот же процесс ревью (pull request), что и код.

**Идемпотентность (idempotency)**
: Свойство операции, при котором повторное выполнение с теми же
параметрами не меняет результат. В API это критично для сетевых
повторов — см. [«Аутентификация» PayFlow API](../api-docs/payment-api/authentication.md).

**Вебхук (webhook)**
: Механизм, при котором сервер сам отправляет HTTP-запрос на заранее
зарегистрированный URL при наступлении события — асинхронная
альтернатива опросу (polling).

**Rate limit (лимит запросов)**
: Ограничение количества запросов к API за единицу времени, обычно
возвращается как ошибка `429 Too Many Requests` с заголовком
`Retry-After`.

## База знаний и поддержка

**KB (Knowledge Base), база знаний**
: Набор статей самообслуживания для пользователей и агентов поддержки,
обычно по шаблону «симптом → шаги → результат».

**Troubleshooting-статья**
: Статья базы знаний, построенная вокруг конкретного симптома или
сообщения об ошибке, а не вокруг задачи «сделать X с нуля».

**SSO (Single Sign-On), единый вход**
: Механизм аутентификации, при котором пользователь входит один раз
через корпоративного identity-провайдера (Okta, Azure AD и т. п.) и
получает доступ сразу к нескольким системам.

## Проблем- и инцидент-менеджмент (ITIL)

**Инцидент (incident)**
: Незапланированное прерывание или снижение качества сервиса. Горизонт
реагирования — минуты и часы.

**Проблема (problem)**
: Первопричина одного или нескольких инцидентов. Горизонт работы — дни
и недели, после того как инцидент уже закрыт.

**KEDB (Known Error Database), база известных ошибок**
: Реестр уже диагностированных проблем с подтверждённым обходным
решением (workaround), который можно применить немедленно.

**RCA (Root Cause Analysis), анализ первопричины**
: Структурированный разбор инцидента, цель которого — найти
первопричину, а не просто зафиксировать симптом. См.
[«Шаблон постмортема»](../problem-management/postmortem-template.md).

**Blameless postmortem, безвиновный постмортем**
: Формат разбора инцидента, в котором фокус — на условиях и системных
факторах, приведших к ошибке, а не на человеке, который её совершил.

**IC (Incident Commander)**
: Единственный человек, принимающий решения в ходе инцидента; координирует
работу, а не чинит систему руками лично.

**SEV (Severity level), уровень серьёзности**
: Классификация инцидента по масштабу влияния (например, SEV1 — сервис
недоступен для всех пользователей). Определяет, кого подключать — см.
[«Матрицу эскалации»](../problem-management/escalation-matrix.md).

**Runbook / плейбук**
: Пошаговая инструкция для стандартной операционной ситуации, написанная
так, чтобы ей можно было следовать под давлением времени, не читая
документ целиком.

## Стандарты документирования (РФ/СНГ)

**ГОСТ 19 (ЕСПД)**
: Единая система программной документации. В основном сохраняет
нумерацию и структуру документов 1970–1980-х годов, носит
рекомендательный, а не обязательный характер.

**ГОСТ 34**
: Комплекс стандартов на автоматизированные системы; существенно
обновлён в 2020–2021 годах, хорошо применим к современным проектам
внедрения ИТ-систем.

**ЕСКД (Единая система конструкторской документации)**
: Комплекс стандартов на конструкторскую документацию — применяется
как референс там, где документация касается технических/инженерных
изделий, а не только программного обеспечения.
