# Чек-листы

Рабочие чек-листы, которые я использую, чтобы документация оставалась
последовательной независимо от того, кто её пишет и когда.

## Чек-лист ревью API-документации

Перед тем как страница или изменение в OpenAPI-спецификации уходит в
релиз:

- [ ] У каждого эндпоинта есть `summary` (короткое) и `description`
      (объясняющее нетривиальное поведение, а не повторяющее `summary`
      другими словами).
- [ ] Разрушительные операции (`DELETE`, необратимые `POST`) явно
      описывают, что происходит с данными — удаляются они безвозвратно
      или архивируются.
- [ ] У каждого параметра и поля тела запроса есть реалистичный `example`,
      а не заглушка вида `string`/`123`.
- [ ] Коды ошибок ссылаются на единый справочник ошибок, а не описаны
      заново на каждой странице.
- [ ] Поля с ограниченным набором значений оформлены как `enum`, а не
      как свободная строка с перечислением в описании.
- [ ] Если добавлен новый обязательный параметр — это отмечено как
      breaking change в changelog.
- [ ] Примеры запросов скопированы и реально выполнены хотя бы один раз
      (curl или аналог) — не просто написаны «на глаз».
- [ ] Спецификация прошла линтинг (например, Spectral) без ошибок
      уровня error.

## Чек-лист публикации статьи базы знаний

- [ ] Заголовок сформулирован как реальный поисковый запрос пользователя
      («Ошибка 429 при импорте задач»), а не как внутреннее название
      фичи.
- [ ] Первый абзац отвечает на вопрос «это про мою ситуацию?» за
      3–5 секунд чтения.
- [ ] Шаги пронумерованы и выполнимы по порядку, без скрытых
      предварительных условий, упомянутых только в середине статьи.
- [ ] Есть раздел «Если не помогло» со ссылкой на следующий шаг
      (эскалация в поддержку, связанная статья), а не тупик.
- [ ] Указано, для какого тарифа/роли/версии продукта актуальна статья,
      если это ограничивает применимость.
- [ ] Статья проверена на актуальность интерфейса (скриншоты, названия
      пунктов меню) в течение последних 2 релизных циклов.
- [ ] Связанные статьи указаны явными ссылками, а не общей фразой
      «см. также документацию».

## Чек-лист перед закрытием инцидента (для IC)

- [ ] Статус в канале инцидента обновлён на «устранено» с точным
      временем.
- [ ] Если SEV1/SEV2 — назначен владелец постмортема и запланирована
      дата разбора (в течение 3 рабочих дней).
- [ ] Временный workaround, если он использовался, зафиксирован в
      [KEDB](../problem-management/known-error-database.md), даже если
      постмортем ещё не проведён.
- [ ] Затронутые пользователи/стейкхолдеры получили финальное
      уведомление о восстановлении сервиса (если получали
      промежуточные).
- [ ] Все, кто был подключён по эскалации, явно оповещены о закрытии
      инцидента.

## Чек-лист публикации release notes

- [ ] Изменения сгруппированы по типу: новое / изменено / исправлено /
      удалено (deprecated).
- [ ] Breaking changes вынесены отдельным блоком в начале, а не
      перемешаны с остальными пунктами.
- [ ] Для каждого пункта есть ссылка на подробности (issue, страница
      документации), если формулировка короче одного предложения.
- [ ] Формулировки написаны с точки зрения пользователя («теперь можно
      X»), а не с точки зрения внутренней реализации («отрефакторили
      модуль Y»).
- [ ] Версия и дата релиза указаны в заголовке в едином формате по
      всей истории release notes.
