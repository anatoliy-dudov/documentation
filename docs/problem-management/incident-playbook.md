# Плейбук реагирования на инциденты

**Аудитория:** дежурный инженер (on-call), получивший алерт по TaskFlow.

!!! danger "Как пользоваться этим документом во время реального инцидента"
    Не читайте документ целиком. Идите по шагам последовательно и
    останавливайтесь на первом, который к вам применим. Разделы
    «Дальнейшее расследование» и «После инцидента» нужны не в первые
    15 минут — не тратьте на них время сейчас.

## Шаг 1 (0–2 минуты). Подтвердите инцидент

- Откройте дашборд **TaskFlow / Overview** в системе мониторинга.
- Проверьте: алерт отражает реальную деградацию (растёт error rate,
  падает доступность) или это одиночный ложный всплеск?
- Если это ложное срабатывание — подтвердите (`acknowledge`) алерт
  с комментарием и завершите работу с этим документом.
- Если инцидент реальный — переходите к шагу 2.

## Шаг 2 (2–5 минут). Определите серьёзность

| Уровень | Критерий | Пример |
|---|---|---|
| **SEV1** | Сервис недоступен для всех пользователей | API возвращает 5xx на все запросы |
| **SEV2** | Сервис деградирован для значимой доли пользователей | Повышенные задержки, ошибки у части регионов |
| **SEV3** | Деградация затрагивает отдельную функцию или небольшую долю трафика | Не работает конкретный редкий эндпоинт |

Уровень серьёзности определяет, кого подключать — см.
[«Матрицу эскалации»](escalation-matrix.md).

## Шаг 3 (5–10 минут). Создайте канал инцидента

- Создайте канал `#incident-YYYY-MM-DD-N` в корпоративном чате.
- Назначьте **Incident Commander (IC)** — по умолчанию это дежурный
  инженер, если не подключился более старший инженер по матрице
  эскалации.
- Опубликуйте первый статус в канале по шаблону:

```text
[SEV{X}] TaskFlow: {краткое описание}
Затронуто: {что именно не работает}
Статус: расследование начато
IC: {имя}
Следующее обновление: через 15 минут
```

## Шаг 4 (10–15 минут). Стабилизируйте сервис

Приоритет — **восстановить сервис**, а не найти первопричину. Типичные
быстрые действия (в порядке возрастания инвазивности):

1. Откатить последний деплой, если инцидент начался в течение часа
   после релиза.
2. Включить feature flag отката проблемной функциональности, если он
   предусмотрен.
3. Масштабировать сервис вручную, если деградация похожа на перегрузку.
4. Только если ничего из вышеперечисленного не подходит — переходить
   к более глубокой диагностике.

!!! warning "Не путайте стабилизацию с починкой"
    Откат деплоя, снятие нагрузки или отключение фичи — это способ
    остановить ущерб, а не найти причину. Реальный разбор причины —
    в постмортеме после закрытия инцидента, а не сейчас.

## Дальнейшее расследование

Если стандартные быстрые действия не помогли — подключайте следующий
уровень по [«Матрице эскалации»](escalation-matrix.md) и проверьте
[«Базу известных ошибок»](known-error-database.md): возможно, симптом
уже встречался и там есть обходное решение.

## После инцидента

- Обновите статус в канале инцидента: `Статус: устранено`.
- Если SEV1 или SEV2 — назначьте владельца постмортема и запланируйте
  разбор в течение 3 рабочих дней, используя
  [«Шаблон постмортема»](postmortem-template.md).
- Если во время инцидента было найдено временное обходное решение —
  зафиксируйте его в [«Базе известных ошибок»](known-error-database.md),
  даже если постмортем ещё не проведён.
