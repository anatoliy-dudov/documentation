# База известных ошибок (KEDB)

**Что это.** Список уже диагностированных проблем с временным
обходным решением (workaround), которое можно применить немедленно,
не дожидаясь постоянного фикса. Это не список открытых багов в трекере —
запись попадает сюда только после того, как найдена рабочая причина
или хотя бы надёжный workaround.

!!! tip "Правило одной записи"
    Каждая запись должна закрывать один конкретный симптом. Если у одной
    первопричины два разных наблюдаемых симптома — заведите две записи
    и свяжите их полем «Связано с», а не описывайте оба симптома в одной
    записи. Так запись находится по тому, что реально видит дежурный,
    а не по внутренней причине, которую он ещё не знает.

## KEDB-042: Повышенная задержка `POST /tasks/bulk` при пакетах > 500 элементов

- **Симптом:** запросы к `POST /tasks/bulk` с более чем 500 задачами
  в пакете превышают таймаут клиента (30 секунд) и возвращают `504`.
- **Первопричина:** пакетная обработка выполняется последовательно внутри
  одной транзакции БД; выше 500 элементов транзакция не укладывается
  в таймаут прокси-сервера.
- **Workaround:** разбивать пакет на части не более 500 задач и отправлять
  несколько последовательных запросов вместо одного большого.
- **Постоянный фикс:** запланирован переход на асинхронную обработку
  пакетов с колбэком по завершении (см. постмортем PM-2026-014).
- **Затронутые версии:** API v1.0–v1.4 (текущая).
- **Статус:** известная проблема, workaround подтверждён.

## KEDB-039: Ложные `payment.failed` вебхуки при таймауте банка-эквайера

- **Симптом:** PayFlow отправляет вебхук `payment.failed`, но при ручной
  проверке через `GET /payments/{id}` статус — `succeeded`.
- **Первопричина:** при таймауте ответа от банка-эквайера платёж
  временно помечается как `failed`, а затем обновляется до `succeeded`
  при получении отложенного ответа от банка — но событие `payment.failed`
  уже успевает уйти подписчикам вебхука.
- **Workaround:** перед необратимыми действиями по `payment.failed`
  (например, отменой заказа) дополнительно проверять актуальный статус
  через `GET /payments/{id}`, а не доверять только событию вебхука.
- **Постоянный фикс:** добавить в вебхук отдельное событие
  `payment.failed.tentative` для таймаутов эквайера, оставив
  `payment.failed` только для окончательных отказов. В разработке.
- **Затронутые версии:** все версии API.
- **Статус:** известная проблема, workaround подтверждён.

## Как пользоваться этой базой во время инцидента

1. Найдите запись по симптому, который вы наблюдаете (Ctrl+F по
   странице), а не по предполагаемой причине.
2. Если запись найдена — примените workaround немедленно, расследование
   первопричины уже не требуется.
3. Если записи нет — после закрытия инцидента заведите новую по этому же
   шаблону, даже если постоянный фикс ещё не готов.
