# Матрица эскалации

Отвечает на один вопрос: **кого подключать и через сколько времени**,
если инцидент не удаётся стабилизировать силами дежурного инженера.
Используется вместе с [«Плейбуком реагирования на инциденты»](incident-playbook.md).

!!! tip "Почему матрица, а не список контактов в мессенджере"
    Список контактов отвечает только на вопрос «кому написать». Матрица
    дополнительно фиксирует **когда** и **при каком условии** — то есть
    убирает необходимость решать это в момент стресса, когда решения
    принимаются хуже, чем в спокойной обстановке.

## Роли

| Роль | Ответственность | Кто обычно |
|---|---|---|
| **Incident Commander (IC)** | Единственный человек, принимающий решения по инциденту; координирует, а не чинит руками | Дежурный инженер по умолчанию; более старший инженер, если подключился |
| **Communications Lead** | Обновления в канале инцидента, статус для стейкхолдеров и (если нужно) публичная страница статуса | Назначается IC на SEV1/SEV2, при SEV3 обычно не нужен отдельно |
| **Subject Matter Expert (SME)** | Эксперт по конкретному компоненту системы, подключается по запросу IC | Владелец соответствующего сервиса |

## Когда и кого подключать

| Условие | Действие | Максимальное время до эскалации |
|---|---|---|
| Инцидент подтверждён, уровень не определён | Дежурный инженер определяет SEV по [плейбуку](incident-playbook.md) | 5 минут с момента подтверждения |
| SEV1 объявлен | Автоматическое уведомление руководителя инженерной команды и SME по затронутому компоненту | Немедленно |
| SEV2, стандартные быстрые действия не помогли за 15 минут | Подключить SME по затронутому компоненту | 15 минут |
| SEV1, не стабилизировано за 20 минут | Подключить руководителя инженерной команды как второго IC-кандидата | 20 минут |
| Любой SEV, требуется публичный статус (внешние пользователи явно затронуты) | Подключить Communications Lead | 15 минут |
| SEV1, не стабилизировано за 45 минут | Подключить руководителя направления (Head of Engineering/CTO) | 45 минут |

## Как передать роль IC

Смена IC — обычная практика при длительном инциденте (например, при
смене часового пояса дежурства), а не признак провала. Порядок передачи:

1. Новый IC явно подтверждает принятие роли в канале инцидента:
   `IC меняется: {старое имя} → {новое имя}`.
2. Предыдущий IC кратко резюмирует статус: что уже проверено, что
   исключено, что делается прямо сейчас.
3. До получения явного подтверждения от нового IC старый IC продолжает
   нести ответственность — роль не считается переданной «по умолчанию».

## Контакты по компонентам

| Компонент | SME / владелец сервиса | Канал для подключения |
|---|---|---|
| TaskFlow API (core) | Команда Platform | `@platform-oncall` |
| PayFlow (платежи) | Команда Payments | `@payments-oncall` |
| Аутентификация / SSO | Команда Identity | `@identity-oncall` |
| Инфраструктура / БД | Команда SRE | `@sre-oncall` |

!!! warning "Держите эту таблицу актуальной"
    Устаревшая матрица эскалации хуже отсутствующей: она создаёт ложную
    уверенность, что нужный человек найдётся автоматически. Ответственный
    за обновление — владелец процесса инцидент-менеджмента, пересмотр —
    не реже раза в квартал и при любой реорганизации команд.
